# challenge-detector — Opus 4.x worklog

## What this lab is for

`hooks/challenge-detector.py` is a Stop hook that challenges the assistant when its final message asserts a cause, state, identity, or quantity as settled fact with no provenance. It fires the question a careful reader would ask — "what makes you think so?" — so the assistant substantiates before the user reads the claim.

The hook's detector is a set of regexes. A regex cannot judge whether a claim is *actually* grounded; it can only match the lexical *shape* of an overconfident claim. This lab measures how far that shape alone gets us, so the choice of which patterns to ship is grounded in numbers rather than intuition, and so the same measurement can be re-run when a new model changes how the assistant phrases its overreach. The folder name carries the model generation (`opus-4-8`) for exactly that reason: this is a per-generation benchmark, expected to be re-run and to drift.

## The data, and how it was built

The reliable marker of this mistake is not in the assistant's wording — it is in the **user's reply**. When the assistant overreaches, the user challenges it with a recognisable family of phrases: "what makes you think", "how do you know", "who said", "where did you get", "what made you", "why do you think". Anchoring on that reply, then reading the assistant turn immediately before it, yields a labelled set of real overreaches without having to guess at the assistant's wording first.

- **Source.** Finished Claude Code sessions under `~/.claude/projects`, searched for those user-challenge phrases in user turns. Sessions where the phrase was quoted text, an ordinary request, or a relative clause rather than a real challenge were discarded. 31 challenged assertions were collected from the surviving sessions; after the Sonnet drop below, 28 remain, drawn from 20 distinct sessions. The full session-ID list with per-assertion model is `manifest.tsv`.
- **Positives** (`positives.txt`): the 28 verbatim assistant assertions that drew a challenge.
- **Negatives** (`negatives.txt`): 200 ordinary assistant sentences from unrelated sessions — the messages a detector must stay silent on.
- **Model sanitisation.** Each challenged assertion was attributed to the model that wrote that specific turn (`.message.model` in the transcript). Three assertions authored by `claude-sonnet-4-6` were dropped, all from two sessions that ran entirely on Sonnet (one of the two produced two of the three). Sessions that mixed models survived: in each, the turn that actually drew the challenge was Opus, so none of the dropped three came from a mixed session. The surviving 28 are Opus: 18 `opus-4-8`, 10 `opus-4-7`. Negatives were likewise restricted to Opus-authored turns. Keeping the model per row (rather than filtering to a single version) lets a stricter `opus-4-8`-only view be taken later without re-mining.
- **Anonymisation.** Real names, emails, places, and business identifiers in the example sentences were replaced with placeholders (`business-a.example`, `Sam Tan`, `Cádiz`, `@fituser`). The placeholder→real map is `key.tsv`, which is git-ignored. Session UUIDs are not personal data and are kept as-is.

## Method

`score.sh` runs each candidate regex (case-insensitive) over both corpora and reports recall (share of the 28 positives matched) and false-positive rate (share of the 200 negatives matched). Replace the two corpora with a new model's data and run it unchanged to reproduce the table. The regexes in `score.sh` are the same strings the hook compiles, so the measured numbers describe the live hook.

## Result

| regex | example matches | recall | FP% |
|---|---|---|---|
| `definite_cause` | "the cause is", "is the reason" | 1/28 | 0% |
| `confident_absence` | "no team", "not documented", "no presence" | 4/28 | 2% |
| `dichotomy_not` | ", not a/the…", "is a real…" | 4/28 | 4% |
| `modal_must` | "cannot", "would conflict", "must be" | 3/28 | 2% |
| `causal_so` | ", so it is / so the user…" | 2/28 | 2% |
| `certainty_adverb` | "unambiguously", "actually", "clearly" | 2/28 | 3% |
| `proof_verb` | "proves", "confirms" | 2/28 | 0% |
| **shipped (OR of the seven)** | | **16/28 = 57%** | **25/200 = 12%** |

The combined false-positive rate (12%) sits above each individual rate because the set is a logical OR: a negative counts once if any regex matches it, so the union is at least the largest single rate and at most the sum of the rates, and need not equal any one of them.

The shipped set misses twelve of the 28. Five of those carry only a number and nothing else, so the rejected `unsourced_number` regex below would catch them (at the false-positive cost that got it rejected). The remaining seven have no lexical marker at all — a cause stated flat ("they were waiting, 30 minutes of low throughput"), a word-number event ("a two-day event"), an identity mapping, an unsourced characterisation. No regex reaches those seven; they are what an LLM judge would be for.

## Rejected candidates

Recorded so the next run can see whether their dynamics change with a new model.

- **`unsourced_number`** (`~N`, `N–M`, `N%`, bare 3+ digit runs). The strongest single catcher (8/28) but the noisiest by far: **19% FP alone**, and including it pushed the combined set to 75% recall / 30% FP. Dropped. The reason it cannot be fixed in regex: its real question is "is this number grounded?", which is semantic. A positional variant — "warn the first time a number appears in the conversation" — fails too, because a number can be both new and correctly sourced (one the assistant just computed and reported with its derivation). This belongs to the LLM judge.
- **Bare copula / definite article alone** ("X is Y", "the X"). Matches almost every declarative sentence; no discriminating power. Not used.
- **Hedges** ("probably", "likely", "might", "I assume", "I think"). Deliberately *not* targeted. These mark uncertainty — the behaviour we want, not the overreach. Flagging them would punish the cure.
- **Generic causal connectors** ("because", "since"). Too frequent in legitimate explanation to carry signal. Not used.

## Why the detector is regex, not a judge

An LLM judge discriminates the bare-declarative misses that no regex can. It is not used because, in this environment, `claude -p` costs about 40 seconds of wall time per call and there is no fast API path, so a judge inside a blocking Stop hook would freeze every turn. A judge therefore has to run out-of-band — fired asynchronously, its verdict delivered on the next turn rather than blocking the current one. That is filed as a follow-up, not built here. The regex set is the synchronous floor that works today.

## Re-running with a new model

1. Re-mine: search finished sessions for the user-challenge phrases, collect the assistant turn before each genuine challenge into `positives.txt`, and a fresh sample of ordinary assistant sentences into `negatives.txt`. Attribute each to its authoring model and keep one model generation; anonymise by extending the existing `key.tsv` (a real identifier that recurs across runs keeps the same placeholder, rather than starting a fresh map per run).
2. Run `./score.sh` (add `--per-line` for the matched-regex breakdown that feeds `manifest.tsv`).
3. Compare the table against this one. A higher miss rate on the shipped set, or a candidate that flips from noisy to clean (or the reverse), is the signal that the new model phrases its overreach differently and the set should be revisited.

## Update — 2026-06-27 19:34 AEST: tightened to three patterns after a live-traffic check

The seven-pattern set scored above (57% recall / 12% FP) was never watched on real traffic before shipping. Replaying the hook over every Opus assistant message from one day of real sessions — 556 messages across seven projects — showed it firing on **37% of them**, three times the benchmark's 12%, and on every project (30–57%), not one kind of work. So the figures in the Result section above are the initial benchmark; the live behaviour and the resulting shipped set are here.

Two reasons the benchmark under-predicted:

- **Unit mismatch.** The benchmark counts false-positives per *sentence* on 200 isolated sentences; the hook fires per *message*, and a message holds many sentences, so the chance one of them trips a pattern is far higher. A 12% per-sentence rate is a ~37% per-message rate.
- **The negatives under-represented dense reasoning.** Real turns are full of correct "X, not Y" distinctions, causal "so", and "must"/"cannot" — careful writing the form cannot tell from overreach.

The fires split sharply by pattern. `confident_absence` and `definite_cause` caught genuine claimed-absences and stated causes (`"email has no such cap" is an assumption I didn't verify`; `the cause is co-channel interference`). The four distinction-and-reasoning patterns — `dichotomy_not` (58 fires), `causal_so` (49), `modal_must` (31), `certainty_adverb` (18) — were 67% of all fires and almost all false (`it's our engagement state, not a fact`; `matched exactly, so it won't hit my shell`).

**Change.** Dropped `dichotomy_not`, `causal_so`, `modal_must`, and `certainty_adverb` from the live hook. The shipped set is now three patterns: `definite_cause`, `confident_absence`, `proof_verb`. `score.sh` keeps the dropped four visible as documented rejects beside `unsourced_number`, so their dynamics stay measurable.

**Result of the change.**
- Benchmark: **25% recall / 2% false-positive** (down from 57% / 12%).
- Live fire rate: **~11%** of messages, 10–19% per project (down from 37%).

The recall cost is real: the three patterns catch about one benchmarked overreach in four. That cost lands squarely on the LLM-judge follow-up (#12), the only thing that reaches the distinction-shaped and bare-declarative cases the regex now leaves alone. The lesson for the next re-run: score false-positives per *message*, not per sentence, and draw negatives from dense real sessions, so the benchmark predicts live behaviour instead of flattering it.
